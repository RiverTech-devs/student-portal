Product vision

Build a Curriculum Network Platform where each node is a curriculum stepping stone, each node can be fed by multiple subjects, and subject-specific trees are really filtered views of one larger learning graph. The macro experience should still feel tree-like and navigable, while the actual engine is graph-based. This matches the strongest direction from your research: macro tree, branch view, detailed rubric view, and student profile layer.

Core design decision

The team needs to treat this as a graph system, not a tree system.

A tree assumes one clean parent-child path. Your model requires:

one node to have multiple incoming dependencies
one node to contribute to multiple future nodes
subject filters that reveal only one slice of a larger structure
developmental and behavioral overlays, not just linear unlocks

That shift is the most important architectural choice in the entire project.

System model
1. One master graph

There is one master curriculum graph for the school.

It contains:

all subjects
all nodes
all dependency links
all progression stages
all observable mastery definitions
all student progress states

Every “Math Tree,” “Science Tree,” or “Life Skills Tree” is just a filtered projection of the same underlying graph.

2. Nodes are curriculum stepping stones

A node is a teachable, assessable competency. It is not just a topic label. It should represent a meaningful “can do” milestone.

Each node must contain:

node ID
node title
primary subject
secondary subject tags
description
grade-band/stage
mastery criteria
evidence types
hard prerequisites
soft dependencies
unlock outputs
visual coordinates
mastery state metadata

This aligns with the research direction that each progression cell should describe mastery in observable behavioral terms, not just list topics.

3. Two dependency types

Every dependency must be labeled as one of two types:

Hard prerequisite
This is required before the node can unlock.

Soft dependency
This does not block unlock, but improves readiness, success likelihood, recommended supports, or confidence.

This solves the earlier problem of trying to make things like responsibility or logic function as strict gatekeepers for every academic node.

4. Three content layers

The team should build the platform around three layers, not one.

Layer A: Graph layer
The real dependency engine.

Layer B: Curriculum layer
Observable mastery, evidence, teacher notes, grade-band descriptors.

Layer C: Student-state layer
Locked, available, in progress, activated, mastered, needs review, freshness, decay, growth trend.

Your current viewer already contains much of Layer C in working form.

Relationship rules

The team should support at least these edge types:

prerequisite_hard
prerequisite_soft
co-requisite
reinforces
transfers_to

That matters because not every connection means the same thing. A “Science supports Math” relation is not the same as “Math must be completed before this node unlocks.”

Progression engine
States

Keep and formalize the states you already have in the viewer:

locked
available
in_progress
activated
mastered
needs_review

Your current HTML already supports this language, including freshness/review behavior and mastery visuals.

Unlock logic

Recommended rule set:

A node becomes available when all hard prerequisites are met.
A node becomes recommended when hard prerequisites are met and soft dependencies are weak.
A node becomes in progress when student work has begun.
A node becomes activated when competency is emerging.
A node becomes mastered when evidence threshold is reached.
A node becomes needs review when mastery decays or freshness drops.
Slope tracking

Track not only current state, but learning trend:

accelerating
steady
plateaued
declining

That comes straight from the research insight that growth trajectory matters, not only current level.

Authoring workflow for curriculum team

The build team should not try to author the whole graph directly in code.

Use a staged authoring pipeline:

Phase 1: Domain drafting

Curriculum leads define subject node lists.

Phase 2: Node normalization

Instructional design team rewrites each node into:

a clear skill statement
stage band
mastery criteria
evidence options
Phase 3: Dependency mapping

Team labels:

hard prereqs
soft supports
cross-domain feeds
Phase 4: Cluster layout

Design team assigns cluster regions and node coordinates.

Phase 5: QA pass

Review for:

dead-end nodes
circular hard dependencies
overlocked pathways
vague mastery definitions
duplicate nodes
Team roles

Owns the philosophy, domain model, and final decisions.

Define nodes, mastery criteria, evidence, and dependency logic.

Graph/data architect

Designs node schema, edge schema, storage model, and query layer.

Frontend engineer

Builds viewer, filters, modals, path highlighting, pan/zoom, and graph rendering.

Backend engineer

Builds APIs, student state engine, progression logic, permissions, and persistence.

UX designer

Designs the filtered subject view, network view, detail cards, teacher workflow, and authoring tools.

QA / curriculum verifier

Tests both technical behavior and academic logic.

Non-negotiable design rules

These should be written into the build brief.

The system is graph-first, not tree-first.
Subject trees are filtered views of one master graph.
Every node must be observable and assessable.
Hard prerequisites must be rare and justified.
Soft dependencies should carry much of the interdisciplinary value.
Character and life skills should usually be overlays or soft feeds, not hard locks.
Every node must point both backward and forward:
what feeds it
what it unlocks
Every node must belong to a stage band.
Every node must have mastery evidence options.
The UI must preserve the wonder and clarity of the current skill-tree viewer.
Risks to avoid

Biggest failure modes:

building disconnected subject trees and only pretending they are connected
making too many hard locks
writing topic labels instead of true competencies
overcomplicating the first release
trying to visually show everything at once
no authoring discipline, leading to messy inconsistent nodes
The handoff summary you can give the team

Use this wording:

We are building a unified K–12 curriculum network.
The system is one master dependency graph, not separate subject trees.
Each node is a curriculum stepping stone with observable mastery criteria.
Nodes may be fed by multiple domains.
Subject trees are filtered visual views of the master graph.
Dependencies must be labeled as hard prerequisites or soft supports.
The UI should preserve the existing constellation/tree feel while exposing deeper cross-domain connections when needed.
The MVP should focus on Math with selected cross-domain links from Logic, Science, and Life Skills.
